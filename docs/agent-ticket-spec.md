# Purpose Reminder Agent Ticket Spec

## 1. 목적
- 이 문서는 "사람이 읽는 업무 요청"이 아니라 "에이전트가 바로 실행 가능한 작업 단위"를 정의한다.
- 티켓 하나는 반드시 `1회 실행 가능한 목표`여야 한다.

## 2. 에이전트 티켓 설계 원칙
1. 티켓은 작은 단위로 쪼갠다. (한 번에 30~120분 내 완료 가능)
2. 결과물 경로를 명시한다. (`어디 파일이 바뀌어야 하는지`)
3. 검증 명령을 명시한다. (`무엇을 실행하면 완료인지`)
4. 중단 조건을 명시한다. (`사람 개입이 필요한 지점`)
5. 수동 작업과 코드 작업을 분리한다.

## 3. 티켓 상태 모델
- `READY`: 실행 가능
- `BLOCKED_MANUAL`: 사람 개입 필요
- `IN_PROGRESS`: 에이전트 수행 중
- `REVIEW`: 코드/문서 결과 검토 필요
- `DONE`: 검증 완료

## 4. 티켓 템플릿 (필수 필드)
```md
## [TICKET-ID] 제목
- 목표:
- 선행 조건:
- 입력 문서:
- 변경 대상:
- 산출물:
- 실행 단계:
1. ...
2. ...
- 검증:
- 완료 기준:
- 중단 조건 (사람 개입 필요):
- 수동 작업:
- 메모:
```

## 5. 사용자(오너)에게 필요한 입력 최소화 규칙
1. 사용자는 "무엇을 만들지"만 말한다.
2. 에이전트는 나머지를 이 문서 + 기존 스펙에서 추론해 티켓을 실행한다.
3. 사람이 꼭 해야 하는 일만 질문으로 올린다.

## 6. 중단 조건 예시
- Apple Developer 로그인 필요
- Signing/Entitlement 설정 화면 접근 필요
- 외부 서비스 콘솔 로그인(API 키 발급) 필요
- 실기기 수동 테스트 필요

## 7. 에이전트 출력 형식 규칙
1. 변경 파일 목록 (절대 경로)
2. 실행한 검증 명령과 결과 요약
3. 남은 리스크
4. 다음 티켓 추천 1~3개
